1. Authentication Flow

Send OTP

Initiates OTP delivery via MSG91 WhatsApp/SMS.

POST /V1/customer/sendOtpOnMobileNumber
Field Type Description
mobile_number String 10-digit number
Rate Limiting & Security (Progressive Delay):

• IP Limit: Max 20 OTP requests per 15 minutes per IP address. Exceeding this returns HTTP 400 with message: "Too many requests from your network. Please try again after 15 minutes."
• Mobile Number Limit (Exponential Backoff): 1st, 2nd, and 3rd OTPs are sent immediately. The 4th attempt requires a 2-minute wait, 5th requires 15 minutes, 6th requires 6 hours, and 7th+ requires 24 hours. Early requests return HTTP 400 with a dynamic message (e.g., "Too many OTP requests. Please wait 2 minutes.").
Exact Response
{
    "code": 200,
    "status": true,
    "message": "OTP sent successfully.",
    "data": {
        "otp_expiration_time": 0,
        "resend_otp_time": null,
        "resend_otp_max_limit": null,
        "is_new_user": true
    }
}
💡 Frontend Developer Note: Login & Referral Flow

Do not ask for the referral code on the very first screen before the mobile number is submitted, because the app does not know if the user is new or existing.

Standard 2-Step Implementation:
  • Step 1: On the Login screen, ONLY ask for the Mobile Number and hit Continue to call sendOtpOnMobileNumber.
  • Step 2: Check the is_new_user variable returned in the response:
    • If is_new_user: true, show the "Do you have a referral code?" text box on the OTP Verification screen (or a separate Profile Setup screen).
    • If is_new_user: false, the user is already registered. Hide the referral code field completely and proceed with simple OTP login.

POST /V1/customer/getServiceSearchData
Field Type Description

category_id Integer Optional. Set this to get Sub-Categories.

Creates a new appointment booking using the new Service Master catalog (supports service variants and dynamic duration calculation).

Field Type Description
city_id Integer Optional. ID of the selected city. If `0` is passed, it defaults to Ahmedabad. If completely empty and logged in, it takes the profile's city.
first_name String Optional. Customer's first name (taken from profile if empty and logged in).
phone String Optional. Customer's phone number (taken from profile if empty and logged in).
appointment_date String Optional. YYYY-MM-DD.
appointment_time String Optional. HH:MM AM/PM.
service_address Integer Required. The ID of the saved address (`user_address_id`) where the service is to be delivered.
items Array of Objects Required. Array of selected services/variants. Each object must contain `service_master_id` and `qty`. Include `variant_id` if a variant was selected. Example: `[{"service_master_id": 10, "variant_id": 5, "qty": 1}, {"service_master_id": 12, "qty": 1}]`
use_wallet_balance Boolean Optional. Set to true/1 if the user wants to apply their wallet balance towards the booking.
payment_type String Required. Must be `"online"` or `"cash"`. If "cash", it skips generating Razorpay Order ID.
💳 Online Payment & Razorpay Flow (Important)
When a user selects "Online Payment" (payment_type: "online"), the backend initiates the online payment process. Here is the strict integration flow:
  1. Step 1 (Create Booking): Frontend calls this bookAppointmentForApp API. The backend saves the appointment with user_payment_status = 'pending'.
  2. Step 2 (Razorpay Order): If successful, this API returns a razorpay_order_id in the response.
  3. Step 3 (Payment Popup): Frontend opens the Razorpay SDK screen using this razorpay_order_id so the user can enter UPI/Card details.
  4. Step 4 (Success Verification): Upon successful payment in the SDK, Razorpay gives 3 keys: razorpay_payment_id, razorpay_order_id, and razorpay_signature. Frontend MUST call the verifyRazorpayPayment API with these 3 keys to mark the order as "Paid".
  5. Step 5 (Payment Failure): If the payment fails on the Razorpay screen or the user closes the popup, do not delete the booking. The booking remains saved in the DB. In the "My Bookings" screen API, you will receive user_payment_status: "pending". Show a "Retry Payment" button to allow them to re-initiate payment. If the Razorpay API itself is down during booking, this API will return an error and set user_payment_status: "failed".
💡 Frontend Developer Note: Wallet Logic Calculation (Checkout Page)
If use_wallet_balance is true, the backend automatically deducts the exact amount and applies it to the final bill. If you want to show the exact discounted amount to the user on the UI before they hit book, calculate it as follows:

Max Usable Limit = (Subtotal * 20) / 100 // 20% limit
Wallet Discount Applied = Min(Wallet Balance, Max Usable Limit)
Grand Total = Subtotal - Coupon Discount - Wallet Discount Applied

Note: Do not call any separate API. Simply calculate this locally and pass use_wallet_balance: true to this booking API.

Verify Razorpay Payment

POST /V1/customer/verifyRazorpayPayment Token Required

Verifies the payment signature returned by the Razorpay mobile SDK and updates the appointment status to Paid/Confirmed.

💳 Mobile Developer Guide: When & How to use this API?
  1. When the user clicks "Book Now" in the app, you first call the bookAppointmentForApp API with "payment_type": "online".
  2. The backend will return a razorpay_order_id (e.g., order_xyz123) and razorpay_key.
  3. You pass this Order ID and Key to the Razorpay Mobile SDK (Checkout popup) to collect the payment from the user.
  4. Once the user pays successfully, the Razorpay SDK returns a success callback with three variables: razorpay_payment_id, razorpay_order_id, and razorpay_signature.
  5. Immediately after the SDK success callback, you must call this verifyRazorpayPayment API and pass those three variables along with the appointment_id. Only then will the backend mark the booking as "Paid"!
Field Type Description
appointment_id Integer Required. The ID of the appointment returned from `bookAppointmentForApp`.
razorpay_order_id String Required. The order ID returned from the Razorpay SDK.
razorpay_payment_id String Required. The payment ID returned from the Razorpay SDK.
razorpay_signature String Required. The signature returned from the Razorpay SDK.
Exact Response
{
    "code": 200,
    "status": true,
    "message": "Payment verified successfully.",
    "data": []
}
